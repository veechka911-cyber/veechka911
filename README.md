# Умная тарелка

Фото-разбор еды без жёсткого подсчёта калорий.

MVP мини-приложения: пользователь загружает фото блюда, выбирает тип еды, отвечает на короткие
уточняющие вопросы и получает примерный информационно-образовательный разбор — баланс тарелки,
КБЖУ, сытность, мягкие подсказки по питанию.

## Что внутри

- React + TypeScript + Vite (статический фронтенд, готов к развёртыванию на российском хостинге).
- Mobile-first, спокойная женственная палитра: молочный, бежевый, мягкий зелёный, оливковый,
  тёплый ягодный.
- 9 экранов: главный, как это работает, загрузка фото, выбор типа блюда, вопросы, результат,
  сбор контакта, продолжение/продажа, история разборов; плюс заглушки политики и пользовательского
  соглашения.
- 6 сценариев разбора: обычная тарелка, суп, салат, перекус, десерт, «не знаю» — у каждого свои
  уточняющие вопросы и своя демо-логика.
- Демо-режим: фото никуда не отправляется, разбор формируется по ответам пользователя.
- История разборов и контакт сохраняются только локально в браузере (`localStorage`).

## Запуск

```bash
npm install
npm run dev      # запуск dev-сервера
npm run build    # сборка production-бандла в ./dist
npm run preview  # локальный предпросмотр собранного бандла
```

## Архитектура и подготовка к продакшену в РФ

Главная функция разбора:

```ts
import { analyzeFoodPhoto } from './src/logic/analyzeFoodPhoto';
```

Сейчас она возвращает результат из локальной демо-логики. В продакшене переключатель `USE_REMOTE`
в `src/logic/analyzeFoodPhoto.ts` направит запрос на `POST /api/analyze-food-photo` — endpoint,
который должен быть развёрнут на backend в РФ. Структура ответа (`AnalysisResult`) одинакова,
поэтому UI не зависит от источника данных.

Целевая архитектура продакшена:

- Frontend — статический бандл из этого репозитория, размещённый на российском хостинге.
- Backend — Node.js или Python FastAPI на сервере в РФ.
- База данных — PostgreSQL на территории РФ.
- Хранилище фото — object storage на территории РФ.
- AI-анализ — российский AI-сервис или локальная vision-модель на сервере в РФ.
- Секретные ключи — только на backend, не в коде фронтенда.

В MVP не используются: Supabase, Firebase, Vercel, Netlify, OpenAI API, зарубежные хранилища и БД
для персональных данных. Фото никуда не отправляется, разбор формируется локально по ответам.

## Юридические рамки

Сервис не является медицинской консультацией, диагностикой, лечением, назначением питания, терапии,
препаратов или БАДов и не заменяет очный приём врача. Все формулировки в интерфейсе подобраны
аккуратно и используются как информационно-образовательные.

## Структура проекта

```
src/
  components/         — переиспользуемые UI-компоненты
  logic/              — типы, вопросы, демо-логика, адаптер analyzeFoodPhoto
  pages/              — 9 экранов + Policy/Terms
  store/              — глобальное состояние (Context) и локальная история
  styles/             — токены палитры и глобальные стили
```
